Interface 1 — varlock load (your main data source)
reads schema → validates → outputs JSON
varlock load --format=json
JSON stdout exit code
What you call it forRead all variables, their resolved values (masked if sensitive), types, validation status, errors
When to call itOn project open · on file change (watch mode) · on manual refresh · after any edit
Key flags--format=json  ·  --env=production  ·  --cwd=/path/to/project
This is the heartbeat of your UI. Every variable row, validation badge, and env summary in your dashboard comes from parsing this command's JSON output. Call it on a file watcher, not on a polling timer.
// stdout from: varlock load --format=json { "env": "development", "valid": false, "errorCount": 1, "warningCount": 0, "variables": [ { "key": "APP_ENV", "value": "development", "type": "enum", "sensitive": false, "required": true, "valid": true, "source": ".env.schema", "errors": [] }, { "key": "OPENAI_API_KEY", "value": "▒▒▒▒▒▒▒▒▒▒▒▒", // always masked "type": "string", "sensitive": true, "required": true, "valid": true, "source": "1password", "errors": [] }, { "key": "DATABASE_URL", "value": null, "type": "url", "sensitive": true, "required": true, "valid": false, "source": null, "errors": ["Required variable is missing"] } ] }
Interface 2 — varlock run (terminal launcher)
spawns child process with env injected
varlock run -- <user command>
streaming stdout/stderr exit code
What you use it forThe "Launch terminal" button. Pipes all output to your xterm.js panel in real time.
Examplesvarlock run -- npm run dev  ·  varlock run -- python manage.py migrate  ·  varlock run -- $SHELL
Env overrideAPP_ENV=production varlock run -- npm run build
How you consume itspawn() with stdio: 'pipe' · pipe stdout+stderr chunks to xterm.js · watch exit code for status
Do NOT use exec() — it buffers. Use spawn() so chunks stream in real time. Pass the user's chosen environment as a process env override before the varlock command. Varlock's startup lines (resolving, validating) appear in the stream before the child process output — show them in the terminal panel as-is.
Interface 3 — varlock scan (security audit)
scans project files for plaintext secret leaks
varlock scan
structured output exit code 0 = clean
What you use it forThe "Scan secrets" button. Shows which files contain plaintext sensitive values.
OutputFile path · line number · which secret key was found · severity
Exit code 0 = no leaks found. Non-zero = leaks detected. Parse stdout for file locations to show in your audit results panel. Surface this prominently — it's a key value-add of the product.
Interface 4 — exit codes (your health signal)
how varlock tells you if things are broken
0
All variables valid — show green status in sidebar project dot
1
Validation errors — required variable missing, type mismatch, secret resolution failed — show red dot
2
Warnings only — optional variable missing, non-critical issue — show amber dot
Interface 5 — filesystem (your write interface)
this is how your UI makes changes — varlock has no write API

Read .env.schema

Your UI reads this file directly using the @env-spec/parser npm library to build the schema editor view. Varlock has no "get schema" endpoint — the file IS the API.

Write .env.schema

When user edits a variable, type, decorator, or description — your UI writes the updated file back to disk, then triggers varlock load to re-validate.

Read .env.* files

Your UI reads .env.development, .env.production etc. directly to show per-environment values. Standard dotenv parsing — no varlock needed for reading these.

Write .env.* files

When user sets a value for a specific environment, your UI writes to the appropriate .env.{env} file. Then re-runs varlock load to confirm validity.
Key insight: Varlock is read-only from the CLI perspective. All mutations (adding variables, changing types, editing values) happen by your UI writing files to disk. Varlock then reads those files on the next load call. Your UI is the writer. Varlock is the reader and validator.
Interface 6 — varlock init (project setup)
one-time migration from .env.example → .env.schema
varlock init
writes .env.schema exit code
What you use it forThe "Add project" wizard step 2 — migrating an existing project from .env.example to varlock
What it doesReads .env.example → infers types → generates .env.schema → writes it to disk
How to call itspawn('varlock', ['init', '--yes'], { cwd: projectPath }) to skip interactive prompts
After this runs, your UI reads the generated .env.schema and shows it in the editor for the user to review and refine. You don't need to parse varlock init output — just check exit code 0 then read the file.
